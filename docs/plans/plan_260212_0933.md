## 0. Coding Standards Update

### One Class Per File Rule

As part of this implementation, we will establish a new coding standard:

> **Each C# class, interface, enum, or record MUST be in its own file.**

**Current Violation:** `VoiceModels.cs` contains 12+ classes/records in a single file.

**Rationale:**

- Improves code navigation and discoverability
- Enables better source control (changes to one class don't show in unrelated file)
- Follows .NET community best practices
- Makes class names match file names (easier to find)

**Action Items:**

1. Update `.github/copilot-instructions.md` with this standard
2. Split `VoiceModels.cs` into individual files
3. Ensure all new code follows this pattern

## 1. Problem Statement

The current NVIDIA Voice Agent implementation runs in **mock mode** when ONNX models are not present locally. Users must manually download models (~3GB total) before the application can perform real ASR transcription. This creates friction in the onboarding experience.

**Current Behavior:**

- Application checks for models at startup
- If not found, logs a warning and enters mock mode
- Mock mode returns simulated transcripts based on audio duration
- No automatic download capability

**Desired Behavior:**

- Application detects missing models on first run
- Automatically downloads required ONNX models from HuggingFace
- Shows download progress in console/log
- Caches models for subsequent runs
- Falls back to mock mode only if downloads fail

## 3. Available .NET Libraries

### Option A: HuggingfaceHub NuGet Package ⭐ Recommended

| Property | Value |
|----------|-------|
| Package | `HuggingfaceHub` |
| Version | 0.1.3+ |
| Downloads | ~13,800 |
| License | Apache-2.0 |
| GitHub | <https://github.com/AsakusaRinne/HuggingfaceHub> |

**Pros:**

- Production-ready, well-maintained
- Built-in progress tracking via `IGroupedProgress`
- Supports resumable downloads
- HuggingFace Hub standard caching (`~/.cache/huggingface/hub`)
- Parallel downloads on .NET 6+

**Cons:**

- Additional dependency
- Requires understanding HF repo structure

**Example Usage:**

```csharp
using HuggingfaceHub;

// Download single file
var modelPath = await HFDownloader.DownloadFileAsync(
    repoId: "onnx-community/parakeet-tdt-0.6b-v2-ONNX",
    filename: "model_int8.onnx",
    localDir: "models/parakeet-tdt-0.6b"
);

// Download with progress
var progress = new GroupedProgress();
progress.OnProgressChanged += (filename, current, total) => 
    Console.WriteLine($"[{filename}] {current}/{total} bytes");

await HFDownloader.DownloadSnapshotAsync(
    repoId: "onnx-community/parakeet-tdt-0.6b-v2-ONNX",
    localDir: "models/parakeet-tdt-0.6b",
    progress: progress
);
```

### Option B: Custom HttpClient Implementation

**Pros:**

- No external dependencies
- Full control over download behavior
- Can customize progress reporting

**Cons:**

- More code to maintain
- Must handle: resume, retries, checksums, rate limiting
- Must implement HuggingFace API integration

**Example Usage:**

```csharp
public class ModelDownloader
{
    public async Task DownloadWithProgressAsync(string url, string destination, 
        IProgress<DownloadProgress> progress, CancellationToken ct = default)
    {
        using var client = new HttpClient();
        using var response = await client.GetAsync(url, HttpCompletionOption.ResponseHeadersRead, ct);
        var totalBytes = response.Content.Headers.ContentLength ?? -1;
        
        await using var source = await response.Content.ReadAsStreamAsync(ct);
        await using var dest = File.Create(destination);
        
        var buffer = new byte[81920];
        long totalRead = 0;
        int bytesRead;
        
        while ((bytesRead = await source.ReadAsync(buffer, ct)) > 0)
        {
            await dest.WriteAsync(buffer.AsMemory(0, bytesRead), ct);
            totalRead += bytesRead;
            progress?.Report(new DownloadProgress(totalRead, totalBytes));
        }
    }
}
```

## 4. Recommended Approach

### Recommendation: Option A (HuggingfaceHub) + Separate Class Library

Create a **new class library** `NvidiaVoiceAgent.ModelHub` that:

1. Uses `HuggingfaceHub` NuGet for HuggingFace downloads
2. Implements `IProgress<T>` for console progress bar
3. Verifies downloads with file size/checksum
4. Can be reused by other projects (CLI tools, other agents)
5. Has no dependency on ASP.NET Core

### Why a Separate Class Library?

| Benefit | Description |
|---------|-------------|
| **Reusability** | Can be used by CLI tools, other web apps, or test projects |
| **Testability** | Easier to unit test without web dependencies |
| **Separation of Concerns** | Download logic isolated from inference logic |
| **Independent Versioning** | Can update models without changing web app |
| **Lighter Dependencies** | Only HuggingfaceHub + logging abstractions |

## 8. Alternative Approaches Considered

### Alternative 1: Pre-bundled Models (NuGet Package)

Create a separate NuGet package containing ONNX models.

**Pros:**

- Simple deployment (`dotnet add package NvidiaVoiceAgent.Models`)
- Versioned with application
- Offline installation possible

**Cons:**

- NuGet size limits (~250MB per package)
- Model split across multiple packages
- Large package restore times
- Licensing complexity

**Verdict:** Not recommended due to size constraints.

### Alternative 2: Docker Image with Models

Include models in Docker image.

**Pros:**

- Single deployment unit
- No runtime downloads
- Reproducible builds

**Cons:**

- 3-4 GB Docker image
- Slow pulls on first deploy
- Not applicable for local development

**Verdict:** Good for production, but doesn't solve local dev experience.

### Alternative 3: CLI Download Tool

Separate `dotnet tool install nvidia-voiceagent-models` command.

**Pros:**

- Explicit user action
- Can be run ahead of time

**Cons:**

- Extra step for users
- Doesn't integrate with application flow

**Verdict:** Could complement auto-download, not replace it.

### Alternative 4: Azure Blob Storage Mirror

Host models on Azure Blob with CDN.

**Pros:**

- Faster downloads (CDN)
- Custom hosting control
- Avoid HuggingFace rate limits

**Cons:**

- Storage costs
- Maintenance burden
- Must keep in sync with HuggingFace

**Verdict:** Consider for high-traffic scenarios.
